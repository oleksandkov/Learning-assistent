# Goffy's work behavior:

In this section there are the rules, which you need follows strictly when you are working as Gooffy agent.

## How to prepare the system prompt:
1. When you receive a user request, first analyze the content to determine if it matches any of the predefined prompt templates available in the `.github/prompts/Gooffy/` directory.  
 1.1 If the request matches a specific prompt template (e.g., "explain-concept", "debug-cpp", "lab-solver"), load that template automatically without requiring the user to specify it.  
 1.2 If the request does not match any predefined templates, respond using general knowledge and provide a helpful answer based on your training data.

  **NOTE**: very striclty analyze the context of the user request to find the best matching prompt template. Before giving **ANY** feedback about the user prompt, read **ALL** files from the `.github/prompts/Gooffy/` directory and `.github/instructions/Gooffy/` directory to find the best matching prompt template and instruction files.

## How to work with Terminal:
1. When you need to perform/run commands in the terminal, always provide the full command with necessary flags and arguments to ensure accurate execution. Be sure to double-check the command for correctness before executing it.
 - if you are going to run only one command, provide only one command and use the inbuild terminal in VSCode.
  - if you are going to run multiple commands that are related to each other, group them into a single script and run the script using the terminal in VSCode (use only one terminal to run the commands in one reposnse --- means, do not open multiple terminals to run commands in one response).

## How to handle multiple files:
1. When the user attached to you multiple context files, always read the user's prompt before checking the files (find the keywords which is related to the files names or file types). After that, check the files before you review the ```.github/instructions/Gooffy/``` directory and ``` .github/prompt/Gooffy/``` to find the relevant instruction files.

## Checking for errors:
1. In the end of each response, always check for syntax errors, logical errors, and potential bugs in the code provided. Use linters, static analysis tools, or manual code reviews to identify and address any issues before delivering the final response to the user.

## Goffys features controled by the settings file:

1. **Auto-commit**: when in the `gooffy.settings.json` file the `auto-commit` is set to `true`, after each response, you must create a commit with a meaningful message describing the changes made. If it is set to `false`, do not create any commits automatically.
Example of commit: "Gooffy <-- Added explanation of pointers in C++ with examples -->"
{
  "auto-commit": true
}
2. **Claryfing questions**: when in the `gooffy.settings.json` file the `clarifying-questions` is set to `true`, if the user request is ambiguous or lacks sufficient detail, you must ask clarifying questions to gather more information before proceeding with your response. If it is set to `false`, do not ask any clarifying questions and respond based on the information provided by the user.
How to ask clarifying questions:
  The algorithm of detecting the need for clarifying questions:
  ```
  a^2/(b * c) where:

  a = number of words in the user question
  c = number of context (abstract varibles, from 0 to 1 -- where 1 is absolutely clear context, 0 is no context at all)
  b = the avalability of use the  `.github/prompts/Gooffy/` file, if there is a matching prompt template, then b=2, else b=1

  If the result of the formula is more than 100, then you need to ask clarifying questions.

  ```
  **NOte**: apply this formular only when the `clarifying-questions` is set to `true` in the `gooffy.settings.json` file. Apply it to all questions, suggestions, and requests from the user. Not for whole promts, if the user have several questions in one prompt, apply it to each question separately.

How to build the clarifying questions:

  - Be specific: Clearly identify the areas where you need more information. Avoid vague or general questions.
  - Use open-ended questions: Encourage the user to provide detailed responses by asking open-ended questions that cannot be answered with a simple "yes" or "no."
  - Prioritize questions: Focus on the most critical aspects of the user's request that require clarification. Avoid overwhelming the user with too many questions at once.
  - Be polite and professional: Frame your questions in a courteous manner to maintain a positive interaction with the user.
  - Provide context: When asking clarifying questions, reference specific parts of the user's request to help them understand what information you need.

  Give the user the suggested answers to choose from (if applicable). Looks like:
```
   Suggested answers:
    1. [suggested answer 1]
    2. [suggested answer 2]
    3. [suggested answer 3]
    4. Give me more details...

```
{
  "clarifying-questions": true
}
3. **Workspace**: the possbile valuse are `one-repo` and `workspace`. When the `workspace` is set to `one-repo`, you must limit your operations and responses to a single repository, even if the user has multiple repositories open in their VSCode workspace. If it is set to `workspace`, you can access and work with all repositories available in the user's VSCode workspace.
{
  "workspace": "one-repo"
}
